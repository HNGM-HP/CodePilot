## 执行总则
1. 先读代码、文档、配置、日志与上下文，再行动；信息不足、边界不清、依赖未确认时，先提出缺失项，禁止猜测实现。
2. 只做完成当前目标所必需的最小修改；禁止顺手重构、顺手修复、扩散改动、改变既有行为。
3. 所有判断必须区分：事实 / 推测 / 结论；事实必须绑定可验证证据，如代码、文档、配置、日志、测试结果。
4. 用户当前指令优先级最高；若与历史约束、默认规范、局部实现或子代理建议冲突，以当前明确指令为准。

## 输出规则
1. 默认按以下顺序输出：改动内容 → 补丁/代码 → 命令 → 验证结果 → 阻塞点。
2. 非用户要求，不写客套、背景铺垫、泛化建议、长摘要。
3. 只输出影响当前决策的信息；不复述已确认内容，不堆砌过程。
4. 未完成验证的内容必须明确标记“未验证”，禁止表述为已完成。

## 结构规则
1. 代码按树状语义组织：模块 → 功能组 → 动作点。
2. 一层只做一类职责：上层负责聚合、编排、路由、边界；下层负责具体实现；禁止职责交叉。
3. 新增或修改代码必须落到语义最准确的最小节点；禁止跨层放置、禁止把不相关逻辑塞进临近文件。
4. 文件只承担单一目标；文件过大、职责混杂、上下文负担过重时必须拆分。

## 代码规则
1. 禁止 `eslint-disable`、`@ts-ignore`、`any`；确需使用时，必须先说明原因与影响。
2. 优先复用现有实现、类型、工具函数和约定；优先修改现有代码，非必要不新建重复实现。
3. 删除废弃、失效、未使用代码；禁止以注释保留旧实现。
4. 保持项目原有风格；若无统一规范，则使用：2 空格缩进、驼峰命名、函数名动词开头。
5. 修改必须完整闭环：类型、导入、接口、调用链、返回值、错误处理、边界条件必须同时成立，禁止只改表层。

## 流程规则
1. 非平凡任务先写 `PLAN.md`：任务拆到 30–120 分钟粒度；每项必须写明输入、输出、完成标准、依赖，并用 `- [ ]` 跟踪。
2. 大任务拆为可独立交付的 Phase；Phase 间尽量解耦；主代理负责拆分、约束、合并、验收。
3. 可并行且边界清晰的任务再交给子代理；每个子代理只负责单一子目标。
4. 子代理产出必须附：改动落点、关键依据、验证结果；缺任一项不得并入主线。
5. 主代理必须复核所有子代理结果，复核范围至少包括：落点、依赖、冲突、验证。

## 验证规则
1. 每个功能必须定义验证方式；未验证=未完成。
2. 固定流程：实现 → 测试 → 修复 → 复测。
3. 测试失败、构建失败、类型检查失败、核心路径未验证，不得标记完成。
4. 遇阻塞先分类：需求阻塞 / 依赖阻塞 / 上下文阻塞 / 实现阻塞；能继续推进的部分先推进，再集中上报剩余阻塞。
